# Fetch registry source from GitHub instead of bundling it

The `fluentui-recipes` CLI no longer ships component/hook **source** inside its
npm package. The published tarball contains only the catalog metadata
(`src/registry/registry.json`) plus `build.json` (the git commit SHA captured at
build time). At `add` time the CLI fetches each file's contents over HTTP from:

```
https://raw.githubusercontent.com/{owner}/{repo}/{sha}/src/registry/{file.path}
```

where `{owner}/{repo}` is parsed from `package.json` `repository.url` and `{sha}`
comes from `build.json`. This supersedes ADR 0001 (bundled local registry).

We chose this to shrink the published package and decouple the shipped artifact
from the recipe source, while keeping installs reproducible: pinning to the exact
build SHA means a given CLI version always fetches the same source.

## Consequences

- `add` now requires network access and is fully async. It is no longer offline
  or hermetic (the trade-off ADR 0001 originally made, now reversed deliberately).
- **The build SHA must be pushed to GitHub before publishing**, or every `add`
  404s. Publish only from a pushed commit (ideally CI after push).
- `build.json` is generated by `scripts/write-build-info.js`, git-ignored, and
  included in the npm tarball via the `files` whitelist. `build:cli` writes it
  before bundling; `prepublishOnly` runs `build:cli` so the SHA is always fresh.
- The CLI fetches all needed files before writing any, so a network/404 failure
  aborts cleanly without leaving a half-installed set.
- `registry.json` and `build.json` are still resolved relative to `dist/cli.js`,
  so that relative layout must stay stable in the published tarball.
